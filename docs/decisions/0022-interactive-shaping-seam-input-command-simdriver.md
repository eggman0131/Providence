# 0022 — The interactive shaping seam: `TerrainCommand`, `SimDriver`, and a core that consumes commands

- **Status:** Accepted
- **Date:** 2026-07-07
- **Deciders:** Director + agent
- **Related:** [`../20-architecture.md`](../20-architecture.md) (§2.4 ports, §4 data flow, §5 boundary rules — I2/I4), [`../30-ai-agent-contract.md`](../30-ai-agent-contract.md) (§5 change classification, I1/I2/I3/I4), [`../40-parameterisation.md`](../40-parameterisation.md) (§2.2 `input.*` / `render.*`), [`0020`](./0020-workbench-runtime-and-rendererport.md) (the `RendererPort` shell whose reserved live-sim seam this realises — esp. §5), [`0019`](./0019-foundation-first-terrain-workbench.md) (item 4, the determinism seam: float/ray work at the edge, discrete recorded commands, fixed timestep), [`0017`](./0017-vertex-heightfield-terrain.md) (the height field and the raise/lower cascade the command applies), [`0004`](./0004-deterministic-core-ports-and-adapters.md) (ports & adapters, core depends on ports never adapters), [`0002`](./0002-llm-as-strategic-advisor.md) (record-replay rests on I3), [`0009`](./0009-enforcement-tooling-and-the-gate.md) (the gate); realises issue [#9](https://github.com/eggman0131/providence/issues/9) and issue [#10](https://github.com/eggman0131/providence/issues/10).

## Context

[ADR 0020](./0020-workbench-runtime-and-rendererport.md) fixed the workbench runtime shell: the `wgpu`/`winit` renderer adapter **owns the window event loop**, draws a read-only [`TerrainFrame`](../../crates/ports/src/lib.rs) snapshot handed across `RendererPort`, and keeps the camera as adapter-local view state. It deliberately scoped issue #8 to a **static** world — the camera moves, the land does not — and **reserved** the live-sim seam for later: *"how fresh frames are produced on the fixed timestep and interpolated for display … deferred to #10, which refines this port"* (ADR 0020 §5), with sim-*affecting* input *"out of scope here; it arrives in #9 as a discrete, recorded command through `InputPort`."*

That reserved seam is now due. Issue [#9](https://github.com/eggman0131/providence/issues/9) is the game's defining verb — click land, pull it up or down, watch the cascade ripple — and issue [#10](https://github.com/eggman0131/providence/issues/10) is the determinism work that keeps a live, mutating sim replayable. They share **one** structural problem, and neither can be built without solving it:

1. **Control inversion.** The renderer owns the event loop and blocks in `run()`; a click is delivered *to* the renderer by `winit`. But turning that click into a terrain change needs the core's cascade ([`raise`/`lower`](../../crates/core/src/terrain/shape.rs)), and the boundary DAG ([`xtask/src/boundary.rs`](../../xtask/src/boundary.rs), [ADR 0020 §](./0020-workbench-runtime-and-rendererport.md)) **forbids the renderer from importing the core** — only a derived snapshot may cross. So the renderer cannot itself apply the cascade; it must reach the sim through an interface whose implementation lives outside the adapter.

2. **Determinism must survive a mouse.** A cursor produces continuous float motion at a variable frame rate — the classic way wall-clock/float/frame-rate coupling leaks into the pure integer core (I3). [ADR 0019](./0019-foundation-first-terrain-workbench.md) item 4 set the *principle* (the float/ray work happens at the edge; the sim advances on discrete, **recorded** commands); this ADR makes the **command seam concrete** so record-replay (the LLM-opponent bet, [ADR 0002](./0002-llm-as-strategic-advisor.md)) holds while the land moves in real time.

Today there is no command type, no seam by which input mutates the sim, and no place a recorded terrain session lives. [`RendererPort`](../../crates/ports/src/lib.rs) is a *push* interface (the app hands the renderer a frame); an interactive loop needs the renderer to *pull* fresh snapshots and *push* commands back. This ADR fixes that seam. It does **not** re-decide `wgpu`/`winit` ([ADR 0007](./0007-wgpu-rendering-framework.md)), the terrain model ([ADR 0017](./0017-vertex-heightfield-terrain.md)), or the render-time shell ([ADR 0020](./0020-workbench-runtime-and-rendererport.md)) — it composes them. Tunable interaction and animation choices (which button shapes, how fast a cascade animates) remain **config** (I1) under `input.*` / `render.*`, not decisions for this ADR.

## Decision

We will realise input as a **discrete, recorded `TerrainCommand`** applied by the core, driven through a new **`SimDriver`** port the renderer holds and calls — the renderer never importing the core.

1. **`TerrainCommand` is a ports-owned, plain data DTO.** A `no_std`, zero-dependency enum in [`providence-ports`](../../crates/ports/src/lib.rs) — `Raise { x, y }` / `Lower { x, y }` — carrying integer grid coordinates only. It is the *single* vocabulary for "shape a vertex", shared by every layer: the input edge produces it, the core consumes it, the session records it. It lives in `ports` for the same reason [`TerrainFrame`](../../crates/ports/src/lib.rs) does — it is a value a port hands across a boundary, so defining it in the interface crate keeps every adapter core-free.

2. **The deterministic core consumes commands, so `providence-core` gains a dependency on `providence-ports`.** The core adds a terrain **`World`** (the height field plus its immovable [`FeatureMap`](../../crates/core/src/terrain/feature.rs)) and an **`apply(command) -> ShapeOutcome`** that dispatches a `TerrainCommand` to the existing bounded cascade ([ADR 0017 §3](./0017-vertex-heightfield-terrain.md)), immovable-refusal and all. Because `apply` takes the ports `TerrainCommand`, a **new edge `providence-core → providence-ports` joins the allowed DAG** ([`xtask/src/boundary.rs`](../../xtask/src/boundary.rs)). This is the direction the architecture already documents — *"the core depends on port interfaces, never on a concrete adapter"* ([ADR 0004](./0004-deterministic-core-ports-and-adapters.md), [`20-architecture.md`](../20-architecture.md) §5 rule 1) — realised for the first time. `providence-ports` stays a zero-dependency leaf (it must never import the core), so no cycle is possible and both crates remain `no_std` and external-dependency-free.

3. **The application session implements `SimDriver` and owns the recorded command log.** A new `SimDriver` port in `providence-ports` is the interactive seam:
   - `submit(command)` — the **input entry point** (the concrete form of the `InputPort` issue #9 names): input reaches the sim *only* through this method, as a discrete `TerrainCommand`. It applies the command and **records** it in the session's log.
   - `width()` / `height()` / `heights()` / `revision()` — the **frame-production** reads (the refinement [ADR 0020 §5](./0020-workbench-runtime-and-rendererport.md) reserved): the renderer pulls the current snapshot to draw, and `revision()` bumps whenever the heights change so the renderer knows to animate.

   The `providence-app` session holds the core `World`, a tick counter, and the log `Vec<(tick, TerrainCommand)>`; it implements `SimDriver` (app→ports is already an allowed edge). A recorded session is therefore *seed + params + log* — everything needed to replay bit-for-bit.

4. **The composition root wires it; the renderer drives it through the trait.** The `providence` binary — the only crate permitted to name concrete adapters ([`20-architecture.md`](../20-architecture.md) §5.2) — builds the session and passes `&mut session` into the renderer's run loop as `&mut dyn SimDriver`. The renderer imports only the *trait* from `providence-ports`; the implementation lives in the app. `RendererPort::present` is **unchanged** — it still seeds the initial frame and serves the static headless/no-op adapters — and the interactive `SimDriver` pull path sits alongside it, exactly the "refines, not replaces" shape ADR 0020 §5 promised.

5. **Commands apply on submit; the recorded log replays wall-clock-free; animation is render-only.** Terrain has no per-tick background evolution yet (water, erosion — parked, [ADR 0019](./0019-foundation-first-terrain-workbench.md)), so each `submit` applies its command and advances the logical tick by one; the log is a wall-clock-free `(tick, command)` sequence that replays by stepping tick-by-tick. **No wall-clock, float, or frame-rate value ever enters the core** (I3). The *only* real-time input in the whole path is the renderer interpolating a vertex's **visual** height from its old integer value to its new one for smooth animation — pure presentation, adapter-local, exactly like the camera ([ADR 0020 §3](./0020-workbench-runtime-and-rendererport.md)). When a future subsystem does need per-tick evolution, the seam already supports a wall-clock-**paced** `advance()` accumulator in the renderer loop (real time decides *when* to step, never *what* a step computes); adding it then is not a redesign.

This is an **architectural** change ([contract §5](../30-ai-agent-contract.md)): it introduces the `TerrainCommand` DTO and the `SimDriver` port, and it adds the `providence-core → providence-ports` edge to the enforced dependency DAG. The interaction/animation *tunables* it relies on are config, not code: the shaping bindings under `input.*` (a registered root, [`40-parameterisation.md`](../40-parameterisation.md) §2.2, first keys land here) and the animation timings under `render.animation.*`.

## Player & experience impact

- **Player / experience:** this is the decision that makes the land *respond to your hand*. Click a hillside and pull it up; the stepped cascade ripples outward as you watch, and pushing down mirrors it. It is the single most-used verb of the whole game, made tangible for the first time — and it is the instrument that finally answers the two open Director rulings by *feel*: *"does the stepped model feel right in motion?"* ([#11](https://github.com/eggman0131/providence/issues/11)) and *"which shaping gesture feels best?"* ([#12](https://github.com/eggman0131/providence/issues/12)). Because every gesture — a single click, a drag, a brush — resolves to the same discrete `TerrainCommand`, the Director can try all three on one seam and rule on which feels right, without any of them changing the architecture underneath.
- **Future flexibility:** the discrete-recorded-command path is the exact seam every later interactive and AI capability reuses — powers that reshape terrain, and eventually the rival deity's moves — all become `TerrainCommand`-shaped writes over the same replayable log. Fixing it now, minimally, keeps the record-replay reproducibility the LLM-opponent design rests on ([ADR 0002](./0002-llm-as-strategic-advisor.md)) intact *while* the world becomes real-time and mutable. And because the renderer still only ever sees a derived snapshot, the terrain model underneath ([ADR 0017](./0017-vertex-heightfield-terrain.md)) stays free to change while it is being judged.
- **What it forecloses:** input is now, by construction, *only* a discrete command — there is no path by which a drag writes a float height or a frame rate changes an outcome. That is the intended constraint (it is what protects determinism), not a limitation to route around: a gesture that wants finer control emits *more* commands, never a continuous one.

## Consequences

- **Positive:**
  - One command vocabulary (`TerrainCommand`) shared end-to-end — input edge, core `apply`, recorded log, replay test — with no duplicate types and no translation site.
  - The renderer stays core-free and holds no game state: it submits commands and pulls snapshots through a trait, so the boundary (I2/I4) and determinism (I3) hold by construction, not by discipline.
  - #9 (the felt interaction) and #10 (the replayable seam) build on **one** foundation instead of inventing two; #10's replay golden and #9's animation are the two faces of the same `SimDriver`.
  - `RendererPort` is untouched, so the static headless-PNG and no-op adapters keep working exactly as before; the interactive path is purely additive.
- **Negative / trade-offs:**
  - The `providence-core → providence-ports` edge is new. It is the documented-intended direction ([ADR 0004](./0004-deterministic-core-ports-and-adapters.md)) and cannot form a cycle (ports is a zero-dep leaf), but it *is* a first: the core now names a ports type. Mitigated by keeping what crosses to plain data (`TerrainCommand`), never a trait the core calls outward.
  - The seam is fixed before drag-sculpt/brush ([#12](https://github.com/eggman0131/providence/issues/12)) are chosen; if a gesture wanted something other than discrete vertex commands the seam would need revisiting. Judged safe — the issue itself states all three candidate gestures resolve to discrete commands.
  - Commands-apply-on-submit is a deliberate simplification of the fixed-timestep loop (one tick per command) valid only while nothing evolves per tick; the wall-clock-paced form is reserved, not built (contract §4.1, no speculative abstraction).
- **Enforcement / gate impact:**
  - **Boundary DAG.** `providence-core`'s allowed edge set gains `providence-ports` in [`xtask/src/boundary.rs`](../../xtask/src/boundary.rs); `providence-ports` stays a leaf with zero edges, and both remain in `ZERO_EXTERNAL_DEPS`. This is the load-bearing change and is reviewed as part of accepting this ADR.
  - **Determinism / replay (I3).** The core replay harness ([`crates/core/tests/replay.rs`](../../crates/core/tests/replay.rs)) gains a golden over a **recorded `(tick, TerrainCommand)` session** on a seeded world, plus a record→replay-identity test — the concrete I3 coverage for a mutating sim (issue #10's deliverable). It extends the existing `GOLDEN_TERRAIN` / `GOLDEN_WORLDGEN` pattern.
  - **Magic-number scan (I1).** `World::apply` is structural command dispatch and adds no behavioural literals to [`crates/core/src`](../../crates/core/src); all interaction/animation numbers live in config. The [scan scope](../../xtask/src/magic.rs) is unchanged.
  - **Config & schema.** New `input.*` authoring structs + a standalone `InputParams` projection, and `render.animation.*` added to `RenderParams`, land with the code that reads them ([ADR 0008](./0008-toml-config-format-types-first-schema.md) sequencing); the generated schema is regenerated (`cargo xtask schema --write`) and the config-key-integrity check keeps them wired.
  - **GPU-free gate (I9).** The pure logic added — command dispatch (core), cursor→command mapping and animation interpolation (renderer) — is unit-tested in the gate; the `winit`/`wgpu` glue stays exercised only through the headless capture path, and the renderer stays coverage-exempt as today ([ADR 0020](./0020-workbench-runtime-and-rendererport.md)).
- **Docs to update (this change, I6):** this ADR + the [`decisions/README.md`](./README.md) index; [`../20-architecture.md`](../20-architecture.md) (`TerrainCommand` / `SimDriver` now concrete in §2.4; the `core → ports` edge in §5; the interactive pull/submit data-flow in §4 alongside the batch/push one); [`../40-parameterisation.md`](../40-parameterisation.md) (the first `input.*` keys and `render.animation.*` land); [`../../CLAUDE.md`](../../CLAUDE.md) (crate-table note that the core now depends on ports for the command DTO, and the interactive workbench). Concrete keys, schema, and `default.toml` entries land **with the code that reads them**, phase by phase.

## Alternatives considered

- **Keep `TerrainCommand` in the core; have the input port carry a ports primitive the composition root translates.** Avoids the new `core → ports` edge, but costs a second command type (a ports mirror) and a translation site in the composition root, and splits the replay vocabulary from the input vocabulary. Rejected in favour of one ports-owned type + one new edge — the edge is the documented-intended direction anyway.
- **Let the renderer import `providence-core` and apply the cascade itself.** Removes the port, but couples the adapter to the core's internal representation and lets a real-time, float-driven, variable-frame-rate loop reach the pure integer sim — precisely the I3 leak this seam exists to prevent. Rejected (it is also already forbidden by the boundary checker).
- **Fold command application into `RendererPort` (add a `submit`/`snapshot` to the existing trait).** Fewer types, but it overloads "present a frame to draw" with "drive the simulation," muddying a port that three adapters (on-screen, headless, no-op) implement — the static ones have no sim to drive. Rejected in favour of a distinct `SimDriver` the renderer *holds*, leaving `RendererPort` focused and the static adapters unchanged.
- **Build the wall-clock-paced fixed-timestep accumulator now.** More complete, but it introduces real-time pacing before any subsystem evolves per tick — speculative machinery (contract §4.1) whose only current effect would be to add a wall-clock surface near the sim. Rejected: commands apply on submit today, and the paced form is reserved for when per-tick evolution actually exists.
- **Continuous (float) drag-sculpt as the primitive.** The most fluid-feeling option in the abstract, but a continuous height write is non-integer and non-replayable — it dismantles I3. Rejected; continuous *gestures* are still available, expressed as streams of discrete commands over the same seam.
